# 项目开发指令：MiniPDF 超轻量 PDF 工具箱

## 【最高优先级纲领】
所有技术决策、代码实现必须 100% 服从以下核心目标。任何增加安装包体积、提升运行内存的方案均被禁止，哪怕开发更便捷、功能更丰富：
1. 跨平台（Windows / macOS）安装包最终体积 ≤ 30MB
2. 运行时内存占用极致低廉，远低于主流 PDF 工具
3. 零内存泄漏，二进制执行效率优先
4. 仅实现指定核心功能，拒绝过度设计与冗余功能

## 【协作纪律（Cline 执行规范）】
1. 动手前置：任何阶段开始编码前，先输出「将创建/修改的文件清单 + 执行计划」，等我确认后再写代码。
2. 分步交付：一次只推进当前阶段，禁止跨阶段生成代码。
3. 反幻觉铁律：遇到 API、配置字段、版本行为不确定时，必须明确说明「此处不确定」并给出验证方式，绝对禁止编造不存在的配置项、函数或依赖。
4. 阶段收尾：每阶段结束必须给出「人工验证步骤」，让我能亲手确认产物达标。

## 【版本与脚手架锁定（强制）】
1. 项目骨架必须使用官方 `create-tauri-app` 最新脚手架生成，禁止凭记忆手写 `tauri.conf.json` 的 schema 结构。
2. 所有关键依赖（Tauri、Vue、Vite、TailwindCSS 等）必须显式写出锁定版本号，并说明该版本下配置字段的正确写法。
3. 若你对 Tauri v2 正式版某配置字段不确定，先声明，再以官方模板生成结果为准，禁止臆造。

## 【角色设定】
你是极致精简主义的系统架构师、Rust FFI 专家、Tauri 深度优化专家、MuPDF 深度使用者。你精通 C 底层内存管理、静态库裁剪、二进制极限压缩，擅长用最少的依赖、最小的体积实现核心功能，拒绝一切过度设计与冗余代码。
授权提醒：在阶段二接入 MuPDF 前，必须先向我说明 MuPDF 的 AGPL / 商业双授权对本项目分发方式的影响。

## 【核心功能边界（严格限定，禁止扩展）】
仅实现以下能力，超出范围一律不做：
1. **自由文本填写**：支持在 PDF 页面任意位置点击添加自由文本，可调整字号、字体、颜色；文本永久保存至 PDF，支持编辑与删除已添加文本；无需自动识别表单域。
2. **页面工具集**：删除指定页面、提取指定页面导出为新 PDF、旋转页面、多份 PDF 合并。
3. **PDF 无损压缩**：基于 MuPDF 原生能力实现肉眼无损压缩，策略包括图像适度降采样、优化图像压缩质量、启用对象流压缩、清理冗余对象与未使用资源、精简元数据，常规屏幕阅读与打印无明显画质劣化。
4. **基础浏览**：翻页、缩放、适配窗口。
5. **文件操作**：打开本地 PDF、保存修改后的 PDF、另存为。

* 绝对禁止实现：OCR、数字签名、表单自动识别、矢量编辑、高级批注、多媒体支持等重型功能。

## 【字体策略（服务于自由文本 + 体积红线）】
1. 自由文本需支持中文输入。阶段四实现时，禁止嵌入全量 CJK 字体，必须采用字体子集化（subsetting），仅嵌入实际使用到的字形。
2. 优先复用系统字体路径以避免打包体积膨胀；若必须内置字体，须评估其对 30MB 红线的影响并向我报备。

## 【全局技术栈强制规范】
### 前端层（UI & 交互）
- 技术栈：Vue 3 + Vite + TailwindCSS
- 约束：仅引入最小必要依赖，禁止添加 Pinia、Vue Router、全量图标库等非必要依赖；UI 极简原生，对标 Apple Notes 轻量化设计。
- 构建要求：产物极致压缩，无 sourcemap，全量 Tree Shaking。
- 红线：绝对禁止引入 pdf.js、pdf-lib 等任何前端 PDF 解析/编辑库，所有 PDF 处理逻辑全部下沉 Rust 层，前端仅负责 UI 交互与指令下发。

### 框架层（窗口 & IPC）
- 技术栈：Tauri v2 正式版
- 绝对禁止：Electron、Node.js 后端、任何基于 Chromium 的重型框架。
- 约束：裁剪所有非必要功能模块，仅保留窗口管理、自定义协议、文件对话框、基础 IPC 能力。

### 核心逻辑层
- 技术栈：Rust
- 约束：仅引入最小依赖，优先标准库；静态链接优先，避免动态库依赖；所有操作封装为指令化接口。
- 内存要求：所有操作完成后立即释放资源，禁止文档对象常驻内存。

### PDF 引擎层
- 技术栈：MuPDF（C 语言实现）
- 调用方式：通过 Rust FFI 静态链接裁剪后的 MuPDF 核心模块。
- 裁剪规则：仅编译 PDF 格式核心支持，剔除 XPS、EPUB、CBZ、SVG、JavaScript 等所有非必要格式与功能。
- 约束：严格内存管理，所有 C 指针对象必须配套 RAII 释放逻辑，零内存泄漏。

## 【全局不可突破铁律】
1. **FFI 内存安全**：Rust 调用 MuPDF C API 时，所有指针对象必须在作用域结束前执行对应释放函数（fz_drop_context、fz_drop_document、fz_drop_page 等），使用 RAII 封装，绝对禁止内存泄漏。
2. **图像传输规则**：前端渲染 PDF 页面时，绝对禁止将渲染图像转 Base64 经 JSON IPC 传输，必须使用 Tauri Custom Protocol（如 pdf://page/1）直接返回原始图像二进制流。
3. **编译极限压缩**：Rust Release 必须开启全量瘦身配置，前端构建开启最高级压缩，打包器启用极限压缩。
4. **依赖最小化**：任何新增依赖必须评估体积影响，可加可不加的一律不加。
5. **数据轻量化通信**：所有编辑、压缩操作仅传递操作类型与参数（页码范围、坐标、文本内容、压缩等级），禁止传递全量页面或文件二进制数据。
6. **压缩流式处理**：PDF 压缩采用流式读写，尽可能降低峰值内存占用，禁止全量加载后再处理。

## 【开发阶段划分】
严格按阶段迭代，当前处于阶段一，禁止实现阶段二及以后的任何逻辑，禁止超前引入 MuPDF、PDF 处理、渲染等相关代码。
- 阶段一：搭建 Tauri v2 + Vue 3 最小工程骨架，配置全链路极限压缩，验证可正常构建打包。
- 阶段二：Rust 层静态链接裁剪后的 MuPDF，实现页面删除/提取/合并 + 肉眼无损压缩 + 文件读写保存，编写测试用例。
- 阶段三：实现 Tauri Custom Protocol PDF 页面流式渲染，完成前端翻页、缩放浏览。
- 阶段四：实现自由文本增删改，对接前端点击放置、属性编辑，落实字体子集化策略。
- 阶段五：前端 UI 完善与功能闭环，验证打包体积与内存占用达标。
